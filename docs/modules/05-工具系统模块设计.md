# 工具系统模块设计

## 1. 模块定位

工具系统定义了模型可执行的能力边界，是“会话决策”到“外部动作执行”的关键层。

主要覆盖：

- `src/tools.ts`
- `src/Tool.ts`
- `src/tools/*`

---

## 2. 职责边界

**负责**

- 维护工具注册清单
- 按模式/权限/开关过滤工具
- 合并内置工具与 MCP 工具
- 对工具调用提供统一约束上下文

**不负责**

- 会话整体编排
- 业务命令解析

---

flowchart LR

---

## 4. 关键设计

## 4.1 统一工具抽象

- 每个工具具备统一的元信息与调用接口；
- 工具执行依赖统一上下文（权限、cwd、状态）。

## 4.2 可见性控制

- 运行模式（如简化模式）会收敛工具集合；
- 权限上下文进一步裁剪可执行范围；
- 特性开关决定实验工具是否暴露。

## 4.3 合并策略

- 内置工具与外部工具统一视图；
- 同名冲突时需要明确优先级策略；
- 合并后保持可观测和可审计。

---

## 5. 工具分类建议（学习视角）

- 文件类：读写编辑、路径搜索
- Shell 类：命令执行与任务管理
- 网络类：Web 搜索与抓取
- 协同类：子任务、提问、待办
- 集成类：MCP 资源与 MCP 工具

---

## 6. 风险与治理

- **工具面过大**：模型选择复杂度上升  
  建议：按场景聚合工具，按风险分层暴露

- **权限策略分散**  
  建议：统一权限决策入口与审计日志格式

- **工具行为不一致**  
  建议：建立工具输入输出规范与错误语义规范
